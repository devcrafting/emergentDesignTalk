Emergent design

Why ? Avoid early choices

Early choices are predictions, but we are really bad at predictions !

Keep options to decide later

How ?

Avoid too much dependencies

Your code will be simpler : KISS ! (Keep It Simple Stupid)

Avoid lots of generic components

You will perhaps not use them : YAGNI ! (You Aren't Gonna Need It)

Avoid components requiring lots of configuration

You can probably do the same with few lines of code

Make small components with few dependencies

Basically what we call hexagonal (or also clean, onion...) architecture

Or also micro-services architecture (new buzz word !)...

When ? At the last responsible moment !

Ie. when you really need to deliver something in production

For demo, you can afford some temporary solutions...

An example of simplicity

Context

Your client ask for an application which will allow to supervise and manage the payment system of a highway toll.

  • Several toll gates, each accepting several payment mode (some may become unavailable)

Great ! Like in any other application :

  • start to design my model
  • generate SQL database structure
  • embed all of this in XXX framework...

No no no !! That's not simplicity ! That is ACCIDENTAL COMPLEXITY !

Start with features, users do not need CRUD !

Describe them with a test, then develop the simplest solution without any implementation detail. You will take care of that later.

Feature 1 : view a summary of all toll gates with number of cars per minute and average revenue by payment mode for each

The simplest solution : a view with a view model representing exactly what you want in this view

Easy to do with TDD + options

  • no matter where data is stored, a simple Repository implementation will do the job when necessary, start with memory implementation
  • no dependency on any model designed upfront
  • caching will be really easy to implement with a so simple Repository

Feature 2 : each time a car arrive at toll gate, propose available payment modes and register payment with one of them

The simplest solution : a view with a view model that gives available payment mode AND a different model registrering payment (emit an event)

Easy to do with TDD + keep options

  • no matter where data is stored, start with a Repository in memory implementation
  • no dependency on any model designed upfront, even the previous one (in feature 1)
  • no matter where this feature is deployed, dependencies does not exist, you could split later in components to deploy separaterly
  • your model could use Event Sourcing with events like "payment registered"

BONUS : you just have done CQRS (Command Query Responsability Segregation)

Wait ! Feature 2 have an impact on Feature 1...but we do not have a common model

That's right ! The simplest solution : an event handler that handles the event payment registered (Feature 2) to update the view model of Feature 1

BONUS : you just have done DDD (Domain Driven Design) with some Bounded Context defined, one for the toll gate payment system, one for the toll supervision

And so on...

Keep things isolated and simple, but refactor as you go to make emerge some simple concepts that will allow you to keep options for later...like perhaps a common Repository, an Event Publisher...

Some tips...

...on the road to software craftsmanship !

Avoid some projects...

...where requirements have been specified at a very detailed level, with lots of implementation details, instead of true user requirement...

Practice your coding skills

With some katas, go to coding dojos...

Open your mind

Participate to communities events, conferences, read blogs...

Go out of your confort zone

Do not rely on your skills, explore other technologies/way of doing things...

Embrace the culture of software craftsmanship !

Read more on this subject...it is a mindset, not a certification or some other marketing thing !

Be careful to avoid classic methodology with emergent design !

Else you could end up with a big bull of mud, because of no refactoring, no code review, no pair programming, quick and dirty falacies...

THAT IS DEFINITELY NOT THE SAME AS EMERGENT DESIGN !
